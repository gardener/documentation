# Decision Log — spec-ci-content-aggregation

## 2026-06-22 — Refinement: edit-link breakage + hugo/public validation

### Issues raised by Niklas

1. **`Edit this page` references `github_repo` + `github_subdir` in frontmatter.** For files originating from `website/`, those fields point at `gardener/documentation/website/...` — a path that ceases to exist the moment PR 2 deletes `website/`. The links 404.
2. The `managed`-flag classifier in the original spec keys on `github_subdir` starting with `website/`. Once we rewrite those fields to fix issue 1, the classifier's heuristic stops firing — they need to be reworked together.
3. **Validate that `hugo/public` is actually unused** — the original `.gitignore` plan kept it, and it might contain load-bearing image assets.

### Investigation

Inspected the working tree, `.vitepress/config.mts`, `.docforge/website.yaml`, and the built `.vitepress/dist/`:

- `.vitepress/config.mts:287-291` builds the edit URL as `${github_repo}/tree/master/${github_subdir}/${fileName}`. Confirmed: the URL goes 404 for locally-maintained files after `website/` removal unless the frontmatter is rewritten.
- `hugo/public/` **does not exist** on disk — it has never existed since the Hugo→VitePress migration. The `.gitignore` entry is dead.
- `website/public/` is the actual source directory for static assets (~2 MB: favicon, logos, og-image, etc.).
- `.docforge/website.yaml:28-31` declares `dir: public / fileTree: ../website/public`, which aggregates `website/public/*` → `hugo/content/public/*` on every Docforge run.
- VitePress's default `publicDir` is `<srcDir>/public` = **`hugo/content/public/`** — verified against `.vitepress/dist/` which contains the favicon, logos, etc. served from that path.
- `.vitepress/config.mts:400` hardcodes the og-image URL as `https://raw.githubusercontent.com/gardener/documentation/refs/heads/master/website/public/og-gardener.png` — also breaks at PR 2.

### Decisions

**D1. Replace `managed: true`-only marker with explicit `local: true` / `managed: true` pair, seeded once in PR 1.**

- The original heuristic ("locally-maintained iff `github_subdir` starts with `website/`") only works on the first PR 1 aggregation, because that's the last moment Docforge stamps those values for ex-`website/` files.
- After PR 1, the file's classification is recorded explicitly as `local: true` or `managed: true`. The classifier reads the marker; it does not re-derive from upstream fields.
- This decouples the steady-state classifier from any future Docforge frontmatter-shape change, and from the `website/` path string.
- Cost: every file gets one extra frontmatter key. Acceptable.

**D2. Rewrite `github_repo` + `github_subdir` on locally-maintained files to point at `gardener/documentation` + `hugo/content/<relpath>`.**

- Reuses the existing `.vitepress/config.mts` URL builder unchanged.
- The rewrite happens in `part-managed.js` (same phase as classification) and is idempotent — re-running yields no diff.
- Considered alternative: drop the fields and hide the Edit link. Rejected — Niklas prefers preserving the affordance.
- Also drop any inherited `path_base_for_github_subdir.to` override, which would otherwise misdirect the URL builder.

**D3. `hugo/public` removed from `.gitignore`; verified it has no production role.**

- Static assets live under `hugo/content/public/` (VitePress `publicDir` convention).
- After PR 1 commits `hugo/content/`, those assets exist in git directly. After PR 2 removes the `dir: public` Docforge block, nothing regenerates them — they sit as locally-maintained files like the rest of the committed content.
- No file move required for PR 2. Niklas's instinct was correct: committing `hugo/content/` in PR 1 captures `hugo/content/public/` along with the markdown.

**D4. Asset relocation to `.vitepress/public/` deferred to Future Work.**

- Niklas's call. Out of scope for this spec; valuable cleanup but independent of the CI-aggregation goal.

**D5. Fix hardcoded og-image URL in PR 2.**

- Update `.vitepress/config.mts:400` to point at `master/hugo/content/public/og-gardener.png`. Without this, social previews break the day `website/` is deleted.

## 2026-06-22 — Pre-flight blocker: org-policy locks Workflow permissions

### Issue

During PR 1 rollout, the maintainer hit `Settings → Actions → General → Workflow permissions` and found both controls **greyed out**: the "Read and write permissions" radio cannot be selected, and "Allow GitHub Actions to create and approve pull requests" cannot be checked. This is enforced at **organisation level** by `gardener` org policy and overrides repo-level configuration.

Consequence: the `aggregate-content.yml` workflow shipped in PR 1 currently has no auth path to push the aggregated `hugo/content/` tree back to the repo. AC #1 cannot be satisfied without resolving this.

The second pre-flight (branch-protection bypass for `github-actions[bot]` on `master`) is **not** blocked by org policy — that control is available on the repo Branch-Protection page and lists `Organization and repository administrators` plus `federated-github-access` already.

### Investigation

1. **Cross-org reference check** (kubernetes-documentation, SAP-internal, separately reviewed via redacted sub-agent): the internal aggregation workflows push directly to `master` using the default `GITHUB_TOKEN` plus job-level `permissions: contents: write`. This works for them because their org does not enforce the "Read only" workflow-permissions block. The pattern itself is not transferable — what they rely on is the absence of the very block we are facing.

2. **GitHub App pattern already exists in `gardener` org**:
   - App: `gardener-github-pkg-mngr`
   - Used by: `gardener/gardenctl-v2`, `gardener/diki`, `gardener/gardenlogin` (release / package-manager workflows)
   - Action: `actions/create-github-app-token@bcd2ba49218906704ab6c1aa796996da409d3eb1   # v3.2.0`
   - Secret-naming convention: `GARDENER_GITHUB_WORKFLOW_<PURPOSE>_CLIENT_ID` + `_APP_PRIVATE_KEY`
   - Permissions are requested directly on the token step (e.g. `permission-contents: write`)
   - App token bypasses the org-locked `GITHUB_TOKEN` permission knob because it carries its own scopes.

3. The existing `gardener-github-pkg-mngr` App is scoped to `homebrew-tap` and `chocolatey-packages` repos. Reusing it for `documentation` is technically possible but semantically wrong (`pkg-mngr` ≠ docs aggregation).

### Decision paths (Niklas to drive with org admins)

**Path A — Lift the org-policy block for `gardener/documentation` (preferred).**
- Approach: org-admin ticket asking that `documentation` be exempt from the "Read only workflow permissions" enforcement (or that the policy be changed globally).
- Pro: zero workflow changes; the PR1 implementation ships as-is.
- Con: org admins may refuse on security grounds (defence in depth).
- Status: pursued first by Niklas.

**Path B — Register a new App `gardener-github-docs-aggregator` (fallback if A is denied).**
- Approach: org-admin registers a dedicated App scoped only to `gardener/documentation` with `contents: write`, generates a private key, maintainer stores Client-ID + Private-Key as repo secrets following the org's `GARDENER_GITHUB_WORKFLOW_*` naming convention.
- Workflow change: insert `actions/create-github-app-token@<sha>` as the first job step, replace `${{ secrets.GITHUB_TOKEN }}` references with the App token output.
- Pro: minimum-scope, self-documenting App name, follows existing org convention.
- Con: highest setup effort (App registration + secret rotation policy).

**Path C — Extend the existing `gardener-github-pkg-mngr` App's installation to also cover `gardener/documentation` (fast fallback).**
- Approach: org-admin adds `documentation` to the App's repository selection, secrets are already in the org.
- Pro: minimal new infrastructure.
- Con: semantically wrong (a docs aggregator using a "package-manager" App); broadens the App's blast radius.

### Decision

**Pursue Path A first.** If denied, fall back to Path B. Path C only if B is also blocked by admin bandwidth.

PR 1 ships unchanged regardless — the workflow is harmless until first dispatch. The first scheduled run (Mon 04:07 UTC after merge, *if* schedule fires at all) will simply fail the push step until auth is resolved. No content damage possible.
